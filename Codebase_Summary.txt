4. Current Service Logic
You have a class extending Service: TriggerMonitorService.kt
onStartCommand method:

override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
    Log.d(TAG, "TriggerMonitorService started")
    when (intent?.action) {
        ACTION_START -> {
            startForegroundService()
            registerAllReceivers()
            initializeScheduling()
        }
        ACTION_STOP -> {
            stopForegroundService()
            return START_NOT_STICKY
        }
    }
    return START_STICKY
}
No AccessibilityService is currently implemented.

5. Trigger Logic
Trigger detection happens in TWO places:
A. BroadcastReceivers (manifest): BatteryReceiver, ConnectivityReceiver, HeadsetReceiver, BootReceiver, TimeAlarmReceiver, GeofenceBroadcastReceiver
B. Foreground Service (TriggerMonitorService): Registers dynamic receivers for battery, screen, airplane mode, Bluetooth, WiFi, network
When a trigger is detected, receivers call workManagerScheduler.scheduleTriggerBasedEvaluation(triggerType)
This enqueues a TriggerBasedWorker which finds all rules with that trigger type and executes them.

Summary of Current Architecture Flow
TRIGGER SOURCES: Manifest Receivers + TriggerMonitorService (Foreground)
Both call WorkManagerScheduler.scheduleTriggerBasedEval(), which enqueues TriggerBasedWorker, which calls ExecuteRuleUseCase (TriggerEvaluator + ActionExecutor)

Key Observations for Architecture Issues
- Duplicate Receivers: Both manifest and service register for same events
- WorkManager Constraints: setRequiresBatteryNotLow(true) may block execution
- Android 13+ Limitations: Many actions just open Settings
- No AccessibilityService: Cannot detect app launches or perform advanced actions
- WakeLock Timeout: Service acquires WakeLock for only 10 minutes
- Manifest Receiver Limitations: On Android 8+, implicit broadcasts may not work reliably in background
